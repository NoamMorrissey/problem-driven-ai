# Conexión con la Fase 4: AI Build

Cuando el Gate Review se completa, el equipo tiene un sistema de Agentes con identidad y límites, Reglas que gobiernan todas las decisiones, y Skills autocontenidos que definen cada tarea. Todo validado, todo trazable.

## La construcción empieza con contexto, no con un briefing

La Fase 4 no empieza con un briefing. Empieza con contexto. Y con un sistema de Agentes, Reglas y Skills de calidad, la construcción tiende a ser la fase de menor coste y menor fricción. El foco no es gestionar la construcción — es proteger la integridad del trípode durante ella.

Cada decisión de un Agente de desarrollo se evalúa contra una única pregunta: **"¿Esta decisión es fiel a las Reglas y al Skill que estoy ejecutando?"** Si lo es, avanza. Si no, se documenta en el Decision Log y se escala al Context Engineer. La construcción no modifica las Reglas silenciosamente. Las respeta o las actualiza explícitamente.

## El contexto como sistema vivo

El contexto no se congela cuando empieza la Fase 4. Los tres elementos continúan evolucionando:

- La construcción revela ambigüedades → se refinan Skills.
- Una decisión local durante un Skill que debería ser global → nueva Regla.
- Un Agente necesita capacidades no previstas → se actualiza la definición del Agente.
- El mercado (Fase 5) cambia la comprensión del problema → se actualizan Reglas y Skills.
- Una asunción activa se refuta → se ajustan las Reglas de restricciones y los Skills afectados.

Cada cambio se documenta en el Decision Log. Cada evolución se traza. El sistema es vivo, pero cada mutación es deliberada y registrada.

## Qué viaja de la Fase 3 a la Fase 4

| Qué viaja | Qué contiene | Cómo lo usa la Fase 4 |
|---|---|---|
| **project-context.md** (Reglas) | Stack tecnológico, convenciones, restricciones, límites de Agentes, protocolo ante gaps, decisiones globales. | Cada Agente de desarrollo lo carga antes de ejecutar cualquier Skill. Es la restricción base. |
| **Definiciones de Agentes** | Identidad, responsabilidades, límites, Reglas que aplican, protocolo ante gaps. | Cada Agente sabe su rol, sus fronteras y qué hacer cuando la información es insuficiente. |
| **PRD** | Qué construir, para quién, casos de uso, prioridades, criterios de éxito. | Guía la secuencia y alcance de los Skills. El QA Agent usa los criterios de éxito para validación. |
| **Architecture Document** | Decisiones técnicas, ADRs, modelo de datos, estrategia de IA, requisitos no funcionales. | Los Agentes de desarrollo aplican las decisiones de arquitectura. Las decisiones del Architect Agent son Reglas. |
| **Story Files** (Skills) | Tareas autocontenidas con objetivo, contexto, criterios de aceptación, dependencias, restricciones. | Cada Skill es una interacción. El Agente de desarrollo lo recibe, aplica Reglas, produce output. |
| **Decision Log** | Cada cambio a Reglas, Agentes o Skills desde su creación. | Trazabilidad. Si surge una pregunta sobre "¿por qué esta regla?", la respuesta está en el log. |

## La cadena de trazabilidad completa

La cadena completa es: **Problem Statement → Solution Brief → PRD → Reglas + Skills.** Si un Skill no puede trazarse hasta el Problem Statement, es invención. Si una Regla no se traza a una decisión de arquitectura, es una asunción. La trazabilidad no es burocracia — es el mecanismo que asegura que la IA construye lo que se acordó, no lo que infiere.

:::tip[Principio #06]
La IA es tan buena como el contexto que recibe. El modelo más potente del mundo con Reglas pobres, Agentes mal definidos y Skills incompletos producirá resultados pobres. Un modelo competente con un sistema de Agentes, Reglas y Skills excelente producirá resultados que ningún equipo humano podría igualar en velocidad ni en consistencia. La diferencia nunca estuvo en la IA. Siempre estuvo en lo que le damos.
:::
