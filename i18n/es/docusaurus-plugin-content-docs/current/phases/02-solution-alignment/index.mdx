# 2. Solution alignment

## El derecho a construir se gana pensando

> *"La mejor forma de tener una buena idea es tener muchas ideas y luego descartar las malas."*
> — Linus Pauling

---

**Qué es:** El proceso iterativo de teorizar soluciones al problema definido, presentarlas a clientes y stakeholders internos, incorporar el feedback recibido y volver a la mesa hasta alcanzar un consenso organizacional tanto en el problema como en la solución. No termina hasta que ese consenso existe.

**Quién participa:** El equipo de producto, los stakeholders internos relevantes (negocio, tecnología, diseño, operaciones según el contexto), y los clientes o usuarios seleccionados para validación. La amplitud del grupo depende del impacto de la solución.

**Qué preguntas se responden:** ¿Qué soluciones son posibles para este problema? ¿Cuál resuelve el problema con mayor precisión y menor fricción? ¿Qué restricciones existen (técnicas, de negocio, regulatorias)? ¿Están todos los stakeholders alineados no solo en la solución sino en el problema que resuelve? ¿Qué asunciones estamos haciendo y cuáles son las más arriesgadas?

**Qué produce:** Un Solution Brief, una descripción clara de la solución propuesta, el problema que resuelve, las decisiones tomadas y sus justificaciones, las asunciones que contiene y los criterios de éxito acordados. Este documento es la base del Context Engineering.

**Cuándo está completa:** Cuando existe consenso organizacional explícito tanto en el problema como en la solución. No cuando hay mayoría. No cuando el que manda dice que sí. Cuando el acuerdo es real y los desacuerdos relevantes han sido resueltos o documentados conscientemente.

:::tip[Principio 03: El consenso es un activo, no un trámite]
**El consenso organizacional real es el activo más valioso que produce esta fase.** No es el Solution Brief, es el acuerdo profundo detrás del Solution Brief. Un Solution Brief sin consenso real es un documento que nadie defenderá cuando las cosas se compliquen.
:::
