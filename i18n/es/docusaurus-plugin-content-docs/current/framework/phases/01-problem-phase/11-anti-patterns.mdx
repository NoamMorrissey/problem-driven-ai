# Anti-patrones: lo que puede salir mal

A lo largo de años de práctica, hemos identificado patrones de fallo recurrentes en el Problem Phase. Conocerlos no garantiza evitarlos, pero aumenta significativamente la probabilidad de detectarlos a tiempo.

:::danger Anti-patrón: El Sesgo Organizado
**Qué es:** El equipo entra al discovery con una hipótesis firme y diseña las entrevistas, consciente o inconscientemente, para confirmarla. Las preguntas son cerradas, los entrevistados se seleccionan porque confirmarán la tesis y los datos que la contradicen se descartan como excepciones.

**Cómo detectarlo:** El Problem Statement final es idéntico a la hipótesis de partida documentada en el kick-off. Ninguna entrevista cambió nada de lo que el equipo creía al empezar.

**Cómo prevenirlo:** Documentar hipótesis antes del discovery. Incluir al menos una pregunta en cada entrevista que busque explícitamente desafiar la hipótesis principal. Asignar a alguien del equipo el rol de "abogado del diablo" durante la síntesis.
:::

:::danger Anti-patrón: El Discovery Express
**Qué es:** El equipo hace 1-2 entrevistas, confirma rápidamente lo que creía y declara el discovery completo en menos de una semana. La presión por avanzar gana a la necesidad de profundizar.

**Cómo detectarlo:** Menos de 5 entrevistas realizadas. El Synthesis Board tiene pocas notas y patrones "obvios". No hay outliers ni tensiones.

**Cómo prevenirlo:** Establecer el mínimo de 5 entrevistas como regla no negociable. Si después de 5 entrevistas no hay tensiones ni sorpresas, probablemente las preguntas no eran las correctas.
:::

:::danger Anti-patrón: El Problem Statement por Comité
**Qué es:** El Problem Statement se redacta en una reunión con demasiada gente, se diluye para que todos estén de acuerdo y termina siendo tan genérico que no dice nada concreto. Es el resultado de optimizar para el consenso político en lugar de para la precisión.

**Cómo detectarlo:** El Problem Statement contiene frases como "necesidad de mejorar la eficiencia operativa" o "optimizar la experiencia del usuario". Si podrías pegar ese Problem Statement en cualquier proyecto de cualquier empresa y seguiría sonando plausible, es demasiado genérico.

**Cómo prevenirlo:** Una persona redacta, el equipo revisa. No se redacta en grupo. El test de especificidad: ¿Este Problem Statement solo es verdad para este problema específico? Si la respuesta es no, hay que ser más preciso.
:::

:::danger Anti-patrón: El Stakeholder Invisible
**Qué es:** Se entrevista a usuarios directos pero se ignora a personas que observan el problema desde otra perspectiva: soporte, ventas de campo, operaciones. El resultado es un Problem Statement correcto pero incompleto.

**Cómo detectarlo:** El Problem Statement solo refleja una perspectiva. No hay tensión entre distintas visiones del problema. El Mapa de Actores tiene roles vacíos.

**Cómo prevenirlo:** Revisar el Mapa de Actores y verificar que al menos un entrevistado representa cada cluster de actores relevante.
:::

:::danger Anti-patrón: La Solución Disfrazada de Problema
**Qué es:** El Problem Statement describe una ausencia de solución en lugar de un problema real. "No tenemos un dashboard de inventario" no es un problema. "Perdemos ventas porque no sabemos cuándo un producto se agota hasta que el cliente lo pide" sí lo es.

**Cómo detectarlo:** Si el Problem Statement contiene la palabra "necesitamos" seguida de una herramienta o funcionalidad, es una solución disfrazada.

**Cómo prevenirlo:** Aplicar el test de la solución múltiple: si el problema está bien definido, debería ser posible imaginar al menos tres soluciones distintas. Si solo admite una, probablemente estás describiendo la solución, no el problema.
:::

:::danger Anti-patrón: El Éxito Sin Definir
**Qué es:** El Problem Statement describe perfectamente la brecha y su coste, pero no dice cómo se verá "resuelto". Cada stakeholder asume una definición de éxito distinta. El equipo construye la solución, la entrega, y tres personas distintas dicen que no es lo que esperaban, cada una por una razón diferente.

**Cómo detectarlo:** No hay criterio de éxito explícito en el Problem Statement. O hay uno tan vago que admite múltiples interpretaciones: "mejorar la visibilidad del inventario" sin definir qué significa "mejorar" ni cómo se medirá.

**Cómo prevenirlo:** Antes de cerrar el Problem Statement, preguntar a cada stakeholder clave: "Si resolvemos esto perfectamente, ¿cómo lo sabrás? ¿Qué será distinto que puedas observar o medir?" Si las respuestas divergen, hay que alinear antes de avanzar.
:::

:::danger Anti-patrón: Las Restricciones Fantasma
**Qué es:** El equipo asume restricciones que no existen ("el sistema no permite eso") o ignora restricciones que sí existen ("seguro que legal no pone problemas"). Ambos errores distorsionan el espacio de soluciones: el primero lo limita innecesariamente, el segundo produce soluciones inviables.

**Cómo detectarlo:** Las restricciones documentadas no han sido verificadas con la fuente. Nadie ha preguntado a IT si realmente el sistema no lo permite. Nadie ha consultado a legal.

**Cómo prevenirlo:** Cada restricción documentada necesita una fuente. "Según el equipo de IT, el ERP no soporta actualizaciones en tiempo real" es una restricción verificada. "El sistema no lo permite" sin fuente es una asunción que puede estar equivocada.
:::
