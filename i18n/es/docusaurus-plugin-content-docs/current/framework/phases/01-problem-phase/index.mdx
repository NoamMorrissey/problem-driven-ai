# 1. Problem phase

Esta es la guía operativa para ejecutar el Problem Phase, la fase en la que el equipo investiga el problema real antes de pensar en soluciones. Mientras que la [sección de metodología](/fases/problem-phase) explica el *por qué* y los fundamentos conceptuales, esta guía cubre el *cómo*: el proceso concreto a seguir, los artefactos a producir, los criterios a cumplir antes de avanzar y los patrones de fallo a vigilar.

## El flujo de Problem Phase

El Problem Phase es un proceso estructurado en cuatro pasos que guían al equipo desde la identificación de un problema hasta su definición validada. Cada paso produce un entregable que alimenta al siguiente: el kick-off establece el mapa de actores y las hipótesis, las entrevistas generan hallazgos en bruto, la síntesis convierte esos hallazgos en patrones, y el Problem Statement destila todo en una definición que tanto usuarios como el Decision Owner reconocen como real. La disciplina en la secuencia es lo que diferencia un discovery riguroso de una conversación bien intencionada.

## Los cuatro artefactos

Esta fase produce cuatro artefactos que juntos forman la base para la Fase 2. Cada uno captura un tipo de conocimiento distinto: quién rodea el problema, qué preguntas hacerle, qué patrones emergen de escuchar, y cuál es la definición destilada del problema real.

1. **Mapa de Actores** — Mapa visual de todos los que tocan el problema: quién lo vive, quién decide si se resuelve, quién se ve afectado por la solución y quién tiene información que el equipo no está viendo.
   **Formato:** Tablero visual colaborativo (Miro, FigJam o pizarra física).
   **Construye:** El equipo completo durante el kick-off.

2. **Interview Guide** — Guía de preguntas abiertas en cuatro bloques (apertura, exploración, profundidad, cierre), incluyendo al menos una pregunta diseñada para desafiar la hipótesis principal del equipo.
   **Formato:** Documento (Google Docs).
   **Construye:** El equipo durante el kick-off, después del Mapa de Actores.

3. **Synthesis Board** — Mapa visual de hallazgos anonimizados agrupados por patrones, con afinidades, tensiones, outliers e hipótesis explícitamente confirmadas o refutadas.
   **Formato:** Tablero visual colaborativo (Miro, FigJam o pizarra física).
   **Construye:** El equipo completo en la sesión de síntesis.

4. **Problem Statement** — Definición del problema de no más de media página con siete elementos: la brecha, el dueño, el criterio de éxito, las restricciones, los actores, el contexto temporal y el coste de la inacción.
   **Formato:** Documento de media página máximo (Google Docs).
   **Construye:** El equipo. Validado por usuarios y Decision Owner.

## El coste del error

¿Por qué invertir en Discovery? Porque corregir un problema mal entendido en producción cuesta **100 veces más** que corregirlo durante Discovery. Cuanto antes se detecte el error, más barata será la corrección.

Esto no es teoría. Es el hallazgo más consistente en la investigación de ingeniería de software: **el coste del cambio crece exponencialmente con cada fase**. Discovery es el momento más barato para equivocarse.

:::tip La inversión que se paga sola
Cada hora invertida en entender bien el problema ahorra semanas de construcción equivocada, ciclos de retrabajo, reuniones de realineación y, sobre todo, la erosión de confianza que se produce cuando un equipo entrega algo que nadie necesitaba. Si estás dedicando menos del 15% del tiempo total del proyecto al Problem Phase, probablemente estás invirtiendo demasiado poco. El punto óptimo está entre el 15% y el 25%.
:::
