# 3. Context phase

Esta es la guía operativa para ejecutar el Context Phase — la fase en la que el equipo traduce la comprensión humana en un sistema que la IA puede procesar fielmente. Mientras que la [sección de metodología](/fases/context-phase) explica el *por qué* y los fundamentos conceptuales (qué son los Agentes, las Reglas y los Skills y por qué importan), esta guía cubre el *cómo*: el proceso concreto a seguir, los artefactos a producir, los criterios a cumplir antes de avanzar y los patrones de fallo a vigilar.

## El flujo de Context Phase

Context Phase traduce la comprensión humana en un sistema que la IA puede procesar fielmente, a través de cinco pasos en secuencia estricta. Las Reglas van primero porque todo lo demás depende de ellas. Los Agentes se diseñan antes que los Skills porque los Skills necesitan saber quién los ejecutará. Los documentos estratégicos van antes de la descomposición porque los Skills se derivan de ellos. La validación va al final porque prueba el sistema completo, no las piezas individuales.

## Los seis artefactos

Esta fase produce seis artefactos que juntos forman el sistema operativo para la Fase 4. A diferencia de las fases anteriores, aquí los artefactos los producen tanto humanos como Agentes de IA, y cada uno tiene productores y consumidores definidos dentro de la cadena.

1. **project-context.md (Reglas)** — La constitución del proyecto. Normas globales que cada Agente carga antes de ejecutar cualquier Skill: stack, convenciones, restricciones, límites de Agentes, protocolo ante gaps, decisiones justificadas.
   **Formato:** Archivo Markdown, 2-5 páginas. Vive en la raíz del proyecto.
   **Construye:** Context Engineer (versión inicial). Architect Agent (actualizaciones).

2. **Definiciones de Agentes** — Fichas de identidad para cada rol de IA: quiénes son, qué hacen, qué no pueden hacer, qué Reglas los gobiernan, qué hacer cuando falta información.
   **Formato:** Un documento Markdown por Agente o documento único con secciones separadas.
   **Construye:** Context Engineer.

3. **PRD (Product Requirements Document)** — Qué construir, para quién, casos de uso, criterios de éxito, prioridades. Traducción del Solution Brief al lenguaje de producto.
   **Formato:** Documento estructurado (Markdown, Google Docs).
   **Construye:** PM Agent a partir del Solution Brief.

4. **Architecture Document** — Decisiones técnicas (ADRs), modelo de datos, estrategia de IA, requisitos no funcionales. Sus decisiones se convierten en nuevas Reglas.
   **Formato:** Documento con ADRs (Architecture Decision Records).
   **Construye:** Architect Agent. Decisiones promovidas a project-context.md.

5. **Story Files (Skills)** — Tareas autocontenidas con objetivo, contexto específico, criterios de aceptación, dependencias y restricciones. Cada Skill es una interacción entre un Agente y su trabajo.
   **Formato:** Un archivo Markdown por Skill, cinco elementos obligatorios.
   **Construye:** SM Agent.

6. **Decision Log** — Cada cambio a Reglas, Agentes o Skills desde su creación: fecha, qué cambió, por qué, quién autorizó, qué artefactos se vieron afectados.
   **Formato:** Registro cronológico (Markdown, hoja de cálculo).
   **Construye:** Todos los participantes. Continuo desde el primer documento.

## Calidad del sistema de contexto

Un sistema de contexto bien diseñado puntúa alto en las cinco dimensiones de calidad. Los gaps en cualquier dimensión producen outputs que parecen correctos pero no son fieles a lo que los stakeholders acordaron.

<RadarChart
  axes={['Fidelidad', 'Completitud', 'Autocontención', 'Trazabilidad', 'Consistencia']}
  datasets={[
    {label: 'Sistema bien diseñado', values: [95, 90, 85, 95, 90], color: 'var(--ifm-color-primary)'},
    {label: 'Sistema con gaps', values: [60, 45, 40, 35, 50], color: 'var(--ifm-color-danger)'},
  ]}
/>

- **Fidelidad** — Los outputs coinciden con el Problem Statement y el Solution Brief. Sin invención, sin reinterpretación.
- **Completitud** — Cada Regla, Agente y Skill tiene todos los elementos obligatorios. Sin contexto implícito.
- **Autocontención** — Un Agente con un Skill + project-context.md puede completar la tarea sin preguntar nada.
- **Trazabilidad** — Problem Statement → Solution Brief → PRD → Reglas + Skills. Cada elemento traza hacia atrás.
- **Consistencia** — Las Reglas no se contradicen entre sí. Los Skills heredan Reglas fielmente. Los Agentes respetan sus límites.

:::tip La inversión del sistema operativo
Si el Context Phase consume más tiempo que la fase de construcción, probablemente estás sobre-especificando Reglas o atomizando Skills en exceso. Si consume menos de un tercio del tiempo de construcción, hay Reglas implícitas y Skills incompletos. La proporción óptima es **30-50% del tiempo de construcción.** Para un proyecto que tardará 4 semanas en construirse, invierte 1-2 semanas en Context Phase. El retorno: la construcción se convierte en la fase de menor fricción.
:::
